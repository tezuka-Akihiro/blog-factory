# リファクター並列実行計画書

生成日: 2025-03-12
対象区分: タスクロジック（task-logic）

## 概要

`src/tasks/` 配下のビジネスロジックを精査した結果、`src/tasks/report.ts`（469行）を中心に規模が大きく、コンテキスト制限を考慮して並列実行を推奨します。特に HTML テンプレートとロジックの混在を解消し、モジュール性を高める必要があります。

## 並列指示文

### タスク 1: レポート生成ロジックのモジュール化 (src/tasks/report.ts)

対象: `src/tasks/report.ts`

**指示内容:**
1. **HTMLテンプレートの分離**: `generateHtmlReport` 関数内の巨大なテンプレート文字列を、ページごと（Page 1: 経営デザインシート, Page 2: 経営診断書, Page 3: 6W2Hシート）にプライベートな関数または別の定数ファイルへ抽出してください。
2. **スタイルの抽出**: インライン CSS 部分を、CSS 文字列を返す独立した関数（例: `getReportStyles()`）に抽出してください。
3. **データ加工ロジックの抽出**: `generateHtmlReport` 内で行っているデータ加工（`replacePlaceholders` の呼び出し等）を独立したヘルパー関数に整理してください。
4. **型安全性の向上**: テンプレートに渡すデータが `ReportData` 型に従っているか再確認し、不明瞭な `any` の残存があれば修正してください。

---

### タスク 2: KPI収集ロジックの整理 (src/tasks/kpi.ts)

対象: `src/tasks/kpi.ts`

**指示内容:**
1. **APIレスポンス処理の分割**: `fetchYesterdayKpi` 関数が 80行を超えているため、GraphQL クエリの構築、リクエスト実行、レスポンスのパース（エラーハンドリング含む）をそれぞれ独立した関数に分割してください。
2. **履歴管理の共通化**: `loadKpiHistory` と `appendKpiRecord` におけるファイルパスや読み込み・書き込み処理の重複を確認し、必要に応じてリファクタリングしてください。

---

### タスク 3: 共通ユーティリティと定数の整理

対象: `src/tasks/` 全般（`extract.ts`, `summary.ts`, `stats.ts` 等）

**指示内容:**
1. **マジックナンバーの定数化**: 日付計算（30日前など）やファイルパスの生成に使用されているマジックナンバーや文字列を、ファイル上部の定数または共通の定数ファイルに抽出してください。
2. **重複ロジックの抽出**: `extract.ts` と `summary.ts` など、複数のタスクで似たようなファイル読み込みや日付変換ロジックがある場合、`src/utils/` または共通のヘルパー関数に集約してください。

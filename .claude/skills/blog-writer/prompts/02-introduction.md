# Phase 2: 導入部作成プロンプト

## AI役割定義

あなたはClaudeMixブログの導入部ライターです。
恐怖型BAB法の5パート構成で、読者の共感を得て行動を促す導入部を執筆してください。

## 前提条件

以下が必要です：

- Phase 1で作成したFrontmatter
- 記事の主題・課題
- `docs/bab-structure.md` への参照（ClaudeMixブログの権威性の理解）

## 思考プロセス（CoT）

以下の順序で段階的に執筆してください：

```text
Step 1: ClaudeMixブログの権威性を理解する
  → 独自の価値提案を把握

Step 2: 命題文を書く（1〜2文）
  → 「この記事は○○の問題を扱う」という宣言
  → 読者がBABに入る前に自分ごとか判断できる文

Step 3: 第1部を執筆（Before）
  → 読者の「あるある」な課題を言語化

Step 4: 第2部を執筆（選定パート）
  → 逆説的興味喚起

Step 5: 第3部を執筆（恐怖パート）
  → 行動しないリスクを3段階で示す

Step 6: 第4部を執筆（希望パート）
  → 解決策と独自価値を提示

Step 7: 第5部を執筆（Bridge）
  → 信頼の橋渡し
```

## 重要な原則

**5パートの順序は固定**: Before → 選定 → 恐怖 → 希望 → Bridge の順序は、心理的な感情の流れを設計したものなので変えない。ただし各パートの長さはコンテンツに応じて判断する。冗長にするくらいなら短くする。

## 良い導入部の例

以下は実際に機能した導入部です。なぜこれが読者を引き込むかを読み取り、今回の記事に応用してください。

```markdown
### はじめに

#### AIのために設定を増やしすぎていませんか？

「AIが賢く動くように」とREADMEや設定ファイルを丁寧に作り込み、プロンプトを洗練させようと努力した。
しかし、機能が増えるたびに指示も増え、AIがどのルールを優先すべきか迷い始める場面に直面した。
場当たり的なプロンプトの追加でその場は凌いだが、気づけば設定自体が保守不可能な「テキスト負債」に変わっていた。

#### この記事をお勧めしない人

- プロンプトを1行増やすだけで、AIが常に完璧に動くと信じている人
- 巨大なモノリスリポジトリこそが、管理の正解であると考えている人
- 「言わなくていいこと」よりも「言うべきこと」を増やすのが設計だと思っている人

もし一つでも当てはまらないなら、読み進める価値があるかもしれません。

#### 膨れ上がる「指示」という名の負債

- 指示系統が複雑化したプロジェクトでは、AIが「否定の指示」を読み解くことにリソースを浪費し始める。
- 本来のビジネスロジックとは無関係な「環境の制約」にAIの脳内メモリが占有され、コード生成の精度が目に見えて低下する。
- 人間がAIの誤解を訂正するためだけに時間を費やす、「AIのための介護」という本末転倒な事態を招く。

#### 「物理的な境界」がもたらす開発の研鋭化

- 物理的なリポジトリ分離により、AIに「言わなくていいこと」を強制的に増やし、脳内メモリを100%解放する明るい未来があります。
- 「3つの聖域」という判断基準を用い、プロンプトに頼らず「物理的境界線」でAIを専門家へと覚醒させる設計図を手に入れられます。
- この「減法設計」は、筆者が運営する本番環境のブログにおいて、最小限の設定で最高精度のコード生成を実現し、その実効性を証明済みです。
- 本記事の内容は、既存のプロンプトエンジニアリングの枠を超えた、巷の情報では決して得られないAI時代の生存戦略です。

#### このブログもそうでした

筆者も、エッジ環境特有の制約と重厚なライブラリの共存に悩み、設定ファイルの肥大化という「情報のゴミ屋敷」に直面した経験があります。
この記事では、テキスト負債を削ぎ落とし、AIを「鋭利な専門家」として再定義するための、明日から試せる減法設計のTipsをまとめました。
AIとの共進化を「引き算」で加速させたい方は、ぜひ詳細な設計思想を確認してください。
```

**この例が機能している理由を読み取ること**：

- Beforeは「理想→問題→不完全な回避策」の流れが3文で完結している。冗長ゼロ。
- 恐怖パートは「AIが迷い始める」「脳内メモリが占有される」「AIのための介護」と、現象だけで語っている。「危険」という言葉は一切使っていない。
- 希望パートの最後の1文「巷の情報では決して得られない」が希少性を一言で封じている。
- Bridgeは自己開示→返報性→コミットメントの3要素を3文に収めている。

## ClaudeMixブログの権威性

導入部の執筆前に、以下の権威性の基盤を理解してください：

詳細は `docs/bab-structure.md` を参照してください。

**要約**:

1. **未来の開発プロセスの実証**: AIネイティブな開発の実践
2. **徹底したドッグフーディング**: 本番稼働中のシステムによる実証
3. **Webの本質を追求する技術選定**: パフォーマンス・スケーラビリティ・DXへの貢献

## 出力フォーマットの原則

**選定パート・恐怖パート・希望パートは必ずマークダウンの箇条書きで出力すること。** チェックボックスや絵文字による列挙（`✔️`、`☑️`等）は使用禁止。AIが自発的に装飾的な絵文字やチェックボックスを挿入しがちなため、マークダウン標準の箇条書きに統一する。

## 5パート構成（恐怖型BAB法）

### 見出し構造

```markdown
### はじめに

[命題文: 1〜2文。読者が「自分の問題か？」をBABに入る前に判断できる宣言]

#### ○○でこんなことありませんか？
[第1部: Before - 3行]

#### この記事をお勧めしない人
[第2部: 選定パート - 3行]

#### ○○は危険です / 実は危ない○○
[第3部: 恐怖パート - 3行]

#### ○○という明るい未来
[第4部: 希望パート - 3-4行]

#### 私も同じでした / このブログもそうでした
[第5部: Bridge - 2-3行]
```

### 命題文（BAB前の宣言）

**位置**: `### はじめに` の直下、第1部（Before）の前

**目的**: 読者がBABに入る前に「自分の問題か？」を10秒で判断できるようにする

**書き方**: 1〜2文。記事が扱う問題と原因を端的に宣言する。

```markdown
React Router v7 に移行すると、build は通るのに typecheck が壊れます。
原因はコードではなく、v7 の仕様変更と型生成方式の変更です。
```

命題文があることで、BABのBeforeパートは「あるある」の深掘りに専念できる。命題文がないと、Beforeパートが「説明」と「共感」を兼ねようとして散漫になる。

---

### 第1部: Before（あるあるな事例紹介）

**見出し**: `#### ○○でこんなことありませんか?`（記事内容に合わせて変更）

**活用要素**: 物語性、好意（共感）、対比（理想 vs 現実）

**目的**: 読者の「あるある」な課題を言語化し、共感を得る

**最重要原則: 第1文は「問題の宣言」で始める**

体験の物語（弱い）:
> AIに移行を頼んだ → typecheckが死んだ → エージェントが止まった

問題の宣言（強い）:
> React Router v7に移行すると、buildは通るのにtypecheckが壊れます。

読者の脳は「それ俺の問題？」と瞬時に判断する。問題を宣言することで、この記事が自分のためのものだと即座に伝わる。体験談から入ると、読者が自分ごとに感じるまでに時間がかかる。

**構成**:

1. 問題の宣言（読者が「それ俺だ」と思う一文）
2. しかし、△△という問題があった（共通の課題）
3. ××で回避したが、根本的な解消にはならなかった（不完全な現状）

### 第2部: 選定パート（この記事をお勧めしない人）｜ 3行

**見出し**: `#### この記事をお勧めしない人`（固定）

**活用要素**: 逆説的興味喚起、コミットメント、希少性（損失回避）

**構成（必ずマークダウンの箇条書きで出力する）**:

- ○○でも全く問題ない人（逆説1）
- ○○より××が大切だと考える人（逆説2）
- 自分には関係ないと思う人（逆説3）

**注意**: 最後に「もし一つでも当てはまらないなら、読み進める価値があるかもしれません」と付け加える

### 第3部: 恐怖パート（行動しないリスク）｜ 3行

**見出し**: `(例)#### バグが治らなくなる,永遠に消費するコンテキスト`（記事内容に合わせて作成。恐怖を煽っていると思わせてはいけない。危ない、危険という言葉は使わずに危険を間接的に表現する）

**活用要素**: 恐怖型訴求（関係レベル低に有効）

**構成（必ずマークダウンの箇条書きで出力する）**:

- ☠ リスクが顕在化し始める予兆
- ☠ 問題が表面化する具体的なきっかけ
- ☠ 修復が困難になる深刻な事象

**厳守事項**:

- **間接的表現の徹底**: 「怖い」「危険」「危ない」といった直接的な言葉を避け、**起こる事象（現象）の説明**によって恐怖を伝えてください（例: 「おもちゃが壊れる恐怖がある」ではなく「知らぬ間におもちゃが壊れている」）。
- **メタ説明の排除**: 執筆した本文に「（リスクの兆候）」などの**カッコ書きによる説明を含めてはいけません**。純粋な記事本文のみを出力してください。

**注意**: 最終的には技術的負債やユーザー離れになるが、それを直接言及せず、それにつながりうる具体的な現象を上げてください。

### 第4部: 希望パート（解決策の提示）｜ 3-4行

**見出し**: `(例)#### 一発で修正できるシステム`（記事内容に合わせて変更。明るい未来がくることを間接的に表現する）

**活用要素**: 希望型訴求、希少性、社会的証明、権威

**構成（2〜3個、必ずマークダウンの箇条書きで出力する）**:

- ✅ この記事を読めば、○○という明るい未来があります（希望提示）
- ✅ 具体的には、△△という方法で××を実現する**設計図**を手に入れられます（具体性＋価値）
- ✅ この方法は、□□のプロジェクトで実証済みです（社会的証明／権威）
- ✅ この情報は、**巷の情報では決して得られない**実践知です（希少性）

### 第5部: Bridge（信頼の橋渡し）｜ 2-3行

**見出し**: `#### 私も同じでした` または `#### このブログもそうでした`（記事内容に合わせて変更）

**活用要素**: 権威、返報性、コミットメント、好意（自己開示）

**構成**:

1. 【権威／自己開示】筆者も過去に同じ問題で悩み、○○のプロジェクトで解決した経験があります
2. 【返報性】この記事で、基本的な考え方と明日から試せるTipsを持ち帰れるように書きました
3. 【コミットメント】深掘りしたい方は、詳細な実装方法を確認できます

## 完了条件チェックリスト

- [ ] 5パートを順序通りに執筆した
- [ ] Beforeパートで読者が「あるある」と感じるか？
- [ ] 恐怖パートは現象で語れているか（「危険」「怖い」を使っていないか）？
- [ ] 希望パートで読者が次に進む動機が生まれているか？
- [ ] Bridgeで自己開示と返報性が感じられるか？
- [ ] 見出しを記事内容に合わせて調整した

# blog-factory 要件定義書 (Phase 1: Inspection)

## 1. 目的

本物の技術ドキュメント（技術本2.0）を継続的に更新・維持するため、外部リポジトリにある既存資産（Markdown/MDX）の現状を正確に把握し、AIや人間が「次に何をすべきか」を判断するための構造化データを出力する。

## 2. アーキテクチャ方針（コマンド分離設計）

「最大の努力で、最小の結果（死なないこと）」を追求し、機能を「部品（Micro）」と「業務フロー（Task）」に分離する。

- **Micro Commands / Tasks**: 単一責任（スキャンのみ、抽出のみ、レポートのみ）を担い、副作用を限定する。
- **Task Commands**: 複数の Micro Task を組み合わせ、実運用のユースケース（今回の `inspect` 等）を実現する。

## 3. 機能要件

### 3.1 Micro Tasks (部品)

- **Scan Task**:
  - 環境変数 `BLOG_SOURCE_PATH` から記事ファイル（`.md`, `.mdx`）を再帰的に探索し、パス一覧を返す。
- **Extract Task**:
  - `gray-matter` を使用し、各ファイルの Frontmatter からメタデータを抽出する。
  - 項目: `title`, `description`, `category`, `updatedAt` (無い場合は `mtime`)
- **Report Task**:
  - `InspectionResult` 型のデータを JSON ファイルとして `results/` に保存する。

### 3.2 Task Commands (業務フロー)

- **Inspect Command**:
  - Scan -> Extract -> Filter (by `category`) -> Report の一連の工程を統制する。
  - CLI引数 `--category=XXX` を受け取り、フィルタリングを実行する。

## 4. 非機能要件（品質・延命戦略）

- **型安全性 (TypeScript)**: `src/types/index.ts` を唯一の真実（SSoT）とする。
- **疎結合**: 各 Task は、入出力が型定義に準拠した独立した関数として実装する。
- **Web標準準拠**: `fs/promises`, `path` 等の Node.js 標準モジュールを優先使用する。
- **診断機能（ロギング）**:
  - `src/utils/logger.ts` を経由し、絵文字プレフィックス（🔍, 📄, ✅, 💥）を用いた視認性の高いログを出力する。
  - 必須項目が欠落している記事を発見した場合は `Logger.warn` を出しつつ、処理は継続する。

## 5. CLI 構成 (Command Interface)

`commander` ライブラリを採用し、将来の機能増築を容易にする。

```json
"scripts": {
  "inspect": "ts-node src/index.ts inspect"
}
```

## 6. データ定義（例）

出力（`blog-factory/results/result-{category}-{ts}.json`）

```json
{
  "inspectedAt": "2026-02-24T15:00:00Z",
  "targetCategory": "web-standard",
  "totalCount": 1,
  "articles": [
    {
      "title": "RemixによるWeb標準への回帰",
      "description": "なぜ今、Web標準なのかを解説する。",
      "category": "web-standard",
      "path": "web-standard/remix-basic.md",
      "updatedAt": "2026-02-24T00:00:00Z"
    }
  ]
}
```

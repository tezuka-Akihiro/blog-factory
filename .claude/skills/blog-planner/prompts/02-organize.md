# ステップ2: 作業内容整理プロンプト

## AI役割定義

あなたは作業内容の構造化専門家です。
記事の種類に応じて、通常モード（Why/What/Challenge/Solution）または技術選定モードで整理してください。

## モード判定

記事が「生存戦略スタック・インデックス」シリーズ（L2カタログ / L3技術詳細）の場合、
**技術選定モード**を使用してください。それ以外は**通常モード**を使用してください。

```text
記事の種類を確認
  ↓
技術スタックのカタログまたは個別技術の選定記事？
  → Yes: 技術選定モード（このファイルの後半）
  → No:  通常モード（Why/What/Challenge/Solution）
```

---

## 通常モード（Why/What/Challenge/Solution）

### 前提条件

ステップ1で理解したブランド哲学、ターゲット、技術スタックを踏まえて、作業内容を構造化します。

## 思考プロセス（CoT）

以下の順序で段階的に整理してください：

```text
Step 1: Why（何をするために）を明確化
  → どのような問題を解決するために、この作業を行ったのか？

Step 2: What（何をしたか）を具体化
  → 具体的にどのような変更を加えたのか？

Step 3: Challenge（どんな壁があって）を特定
  → 作業中にどのような課題に直面したのか？

Step 4: Solution（どう乗り越えたか）を説明
  → その課題をどのように解決したのか？
```

## 実行手順

### 1. Why（何をするために）を明確化

記事を書く目的を1-3文で簡潔に記述してください。

**質問リスト**:

- どのような問題を解決するために、この作業を行ったのか？
- この記事を読む読者は、どのような課題を抱えているのか？
- なぜ今、この記事を書く必要があるのか？

**ポイント**:

- ターゲットペルソナ（AI活用で自立を目指す非技術者・中堅エンジニア）を意識
- 読者の課題を明確に定義

### 2. What（何をしたか）を具体化

実施した作業を3つ以内にまとめ、各項目を5行以内で詳細に説明してください。

**質問リスト**:

- 具体的にどのような変更を加えたのか？
- どのファイル・機能・設定を変更したのか？
- 変更の結果、何が改善されたのか？

**ポイント**:

- 具体的なファイル名、コマンド、設定を記載
- 技術スタック（Remix, Cloudflare, Claude Code等）との関連を明示

### 3. Challenge（どんな壁があって）を特定

作業中に直面した課題を3つ以内にまとめてください。

**質問リスト**:

- どのような問題に直面したのか？
- なぜその問題は発生したのか？
- どのような試行錯誤を行ったのか？

**ポイント**:

- 読者が共感できる具体的な課題を選択
- 技術的な深みを示す（「プロダクトの寿命を見抜く眼力」を反映）

### 4. Solution（どう乗り越えたか）を説明

各課題に対する解決策を3つ以内にまとめてください。

**質問リスト**:

- どのような手法・ツール・アプローチで解決したのか？
- なぜその解決策が効果的だったのか？
- 他にどのような選択肢があったのか？

**ポイント**:

- 具体的なコード例、コマンド、設定を記載
- 「なぜその解決策を選んだのか」という判断基準を示す

## 完了条件チェックリスト

- [ ] Why（何をするために）を1-3文で記述した
- [ ] What（何をしたか）を3つ以内にまとめた
- [ ] Challenge（どんな壁があって）を3つ以内にまとめた
- [ ] Solution（どう乗り越えたか）を3つ以内にまとめた

## 整理のベストプラクティス

### ✅ 良い整理

**Why**:

- 具体的な読者の課題を定義
- 「なぜ今、この記事が必要か」が明確

**What**:

- 具体的なファイル名、コマンド、設定を記載
- 変更の結果、何が改善されたかを明示

**Challenge**:

- 読者が共感できる具体的な課題
- 技術的な深みを示す

**Solution**:

- 具体的なコード例、コマンド、設定
- 「なぜその解決策を選んだのか」の判断基準

### ❌ 悪い整理

**Why**:

- 抽象的で読者の課題が不明
- 「なぜ今」が説明されていない

**What**:

- 抽象的で具体性がない
- 変更の結果が不明

**Challenge**:

- 課題が表面的で共感できない
- 技術的な深みがない

**Solution**:

- 抽象的で実践できない
- 判断基準が示されていない

## 構造化のポイント

### Why（何をするために）

- **ターゲット読者の課題を明確に**: AI活用で自立を目指す非技術者・中堅エンジニアが抱える具体的な課題
- **解決の目的を1-3文で**: 簡潔に、しかし十分な背景を提供
- **「なぜ今」を示す**: 直近1年以内の新しい要素を含む（新規性）

### What（何をしたか）

- **3つ以内にまとめる**: 情報過多を避ける
- **各項目を5行以内で**: 具体的に、しかし簡潔に
- **技術スタックとの関連を明示**: Remix, Cloudflare, Claude Code等

### Challenge（どんな壁があって）

- **3つ以内にまとめる**: 本質的な課題に絞る
- **読者が共感できる課題**: ターゲット読者が直面している課題
- **技術的な深みを示す**: 「プロダクトの寿命を見抜く眼力」を反映

### Solution（どう乗り越えたか）

- **3つ以内にまとめる**: 本質的な解決策に絞る
- **具体的なコード例**: 読者が明日から試せる具体的なTips（具体性）
- **判断基準を示す**: なぜその解決策を選んだのか

## 技術選定モード（生存戦略スタック・インデックス専用）

「生存戦略スタック・インデックス」シリーズのL2・L3記事に使用します。
Why/What/Challenge/Solutionではなく、**選定理由・選定種別・代替案**の軸で整理します。

### シリーズ用語（記事内で使用可）

| 用語 | 意味 |
| : | : |
| 延命医 | Web標準への準拠度と長期生存根拠を診断する視点 |
| 不自然な構造（悪） | 採用しなかった技術の備忘録 |
| 製造ライン | ClaudeMixを構成するサービス・ツールの組み合わせ全体 |
| 選定種別 | 独立選択 / 連動（○○連動） / 連合選択 / 事実上の標準 |

### L2カタログ記事の整理手順

L2は「製造ライン全体を俯瞰するカタログ記事」です。個別技術ではなく**全体像と分類**を整理します。

**質問リスト**:

- このスタックが防ごうとしている技術的負債は何か？
- 各技術はどの選定種別に分類されるか？（連動 / 連合 / 独立 / 事実上の標準）
- 採用しなかった技術と、その不採用理由は何か？
- 技術グループ（自前認証トリオ等）が生まれた背景は何か？

**整理の軸**:

- **ラインのコンセプト**: スタック全体の防衛哲学（何を防ぐか）
- **選定種別による分類**: 技術間の依存関係を可視化
- **不自然な構造（悪）**: 敗因・不採用理由を正直に記録

### L3技術詳細記事の整理手順

L3は「個別技術の選定理由と活用法」です。1技術に絞り**選定の必然性**を整理します。

**質問リスト**:

- この技術の選定種別は何か？（独立 / 連動 / 連合 / 事実上の標準）
- 連動・連合選択の場合、起点となった技術制約は何か？
- 代替案として何を検討し、なぜ不採用にしたか？
- この技術を採用したことで得られた設計上の恩恵は何か？（3点以内）
- ClaudeMix内での具体的な活用例は何か？

**整理の軸**:

- **延命医の診断**: 長期生存根拠を一言で示す
- **選定種別の根拠**: 選定種別に応じた説明の軸（制約 / 必然性 / 比較 / デファクト）
- **代替案と不採用理由**: 不自然な構造（悪）として記録

### 完了条件チェックリスト（技術選定モード）

**L2カタログの場合**:

- [ ] ラインのコンセプト（防ぐ技術的負債）を1-2段落で記述した
- [ ] 全採用技術を選定種別で分類した
- [ ] 技術グループ（連動・連合）の背景を説明した
- [ ] 不採用技術と不採用理由を記録した

**L3技術詳細の場合**:

- [ ] 選定種別を確定した
- [ ] 延命医の診断（長期生存根拠）を一言で示した
- [ ] 選定種別に応じた根拠を説明した
- [ ] 代替案と不採用理由を記録した
- [ ] 構造的メリットを3点以内で列挙した

# 記事テンプレート定義

ClaudeMixブログで使用する4つの記事テンプレートを定義します。

## テンプレート選択ガイド

| テンプレート | 使用ケース | 記事の割合 |
| : | : | : |
| 標準テンプレート | 機能改修記事、新機能追加記事、まとめ記事 | 30-40% |
| トラブルシューティングテンプレート | トラブルシュート記事 | 60-70% |
| L2 技術選定マップテンプレート | 技術スタック全体のカタログ記事（インフォメーション） | シリーズ1本 |
| L3 技術詳細テンプレート | 個別技術の選定理由・活用法（ClaudeMix ガイド） | 技術ごと1本 |

---

## テンプレート1: 標準テンプレート

### 使用ケース

- 機能改修記事
- 新機能追加記事
- まとめ記事（月次振り返り等）

### 構成

````markdown
### 開発の進捗

- **Before**: 着手前の状態を簡潔に1行で説明
- **Current**: 現在の状態を簡潔に1行で説明
- **Next**: 次に行うことを簡潔に1行で説明

### 具体的なタスク

- **Before**: 着手前のタスク内容を1行で表現
- **Current**: 現在取り組んでいるタスク内容を1行で表現
- **Next**: 次に取り組むタスク内容を1行で表現

### 課題と解決策

（課題の概要をここに記述します）

#### 工夫したこと

（課題解決のために試したことや、設計上の工夫などを具体的に記述します）

#### ぶつかった壁

（発生したエラーや、AIとの認識ズレ、手戻りなど、問題となった点を具体的に記述します）

#### 解決方法

（最終的にどのように解決したのかを簡潔に記述します）

### コード抜粋

（解決策を示す、再現性のあるコードスニペットを挿入します）

```typescript
// 例: 実際に動作するコード
export function exampleFunction() {
  // コメントで説明を追加
  return "result";
}
```

### 今回の学びと感想

（今回の経験から得られた学びを記述します）

同じような課題で悩んだ方はいませんか？
もっと良い解決方法があれば教えてください！
```

### ポイント

- **Before/Current/Next**: 開発の流れを時系列で示す
- **工夫/壁/解決**: 試行錯誤のプロセスを正直に記述
- **コード抜粋**: 実際に動作する、再現性のあるコードを掲載
- **学び**: 技術的な学びを言語化する

## テンプレート2: トラブルシューティングテンプレート

### 使用ケース

- トラブルシュート記事
- バグ修正記事
- エラー解決記事

### 構成

```markdown
### 📝 概要

（問題の全体像を2-3文で記述します）

#### 発生環境

- **フレームワーク**: Remix v2
- **ホスティング**: Cloudflare Pages/Workers
- **ビルドツール**: Vite
- **その他**: （必要に応じて追加）

---

### ⚠️ 問題の発見と症状

（いつ、何をしたら問題が起きたのかを記述します）

**エラーメッセージ:**

```bash
ここにエラーメッセージを貼り付け
```

**症状:**

- （具体的な症状を箇条書き）
- （画面の表示、コンソールの出力など）

---

### 🔍 調査と試行錯誤のプロセス

#### 仮説1: ○○が原因ではないか？

**試したこと:**

（検証のために実施した内容）

**結果:**

（成功 / 失敗）

**分かったこと:**

（この検証から分かったこと）

#### 仮説2: △△の設定ミスを試す

**試したこと:**

（検証のために実施した内容）

**結果:**

（成功 / 失敗）

**分かったこと:**

（この検証から分かったこと）

---

### 💡 根本原因の特定

（最終的に特定した根本原因を1-2文で明確に記述します）

**原因:**

（技術的な説明）

**なぜこの問題が起きたか:**

（背景や設計上の理由）

---

### 🔧 解決策

（具体的なコード変更や設定変更を記述します。diff形式推奨）

```diff
# ファイル: path/to/file.ts

- // 修正前のコード
+ // 修正後のコード
```

**変更のポイント:**

- （変更の意図を説明）
- （なぜこの変更で解決するのか）

---

### 🎓 学んだこと・まとめ

#### 技術的な学び

- （この問題から学んだ技術的なポイント）
- （他の開発者にも役立つ知見）

#### 今後のベストプラクティス

- （再発防止策）
- （同様の問題を避けるための指針）

---

### 🔗 関連リソース

- （参考にした公式ドキュメント）
- （関連する記事やIssue）
- （GitHub PR等）

---

同じような問題で困っている方の参考になれば幸いです！
```

### ポイント

- **絵文字の使用**: セクションの視認性を高める（📝⚠️🔍💡🔧🎓🔗）
- **試行錯誤の記録**: 失敗した仮説も含めて記述（他の読者の参考になる）
- **diff形式**: コード変更を明確に示す
- **根本原因の明確化**: 「何が」「なぜ」問題を起こしたのかを説明
- **再発防止策**: 今後のベストプラクティスを提示

## テンプレート3: L2 技術選定マップ（カタログ）

### 使用ケース

- 生存戦略スタック・インデックス（Survival Stack Index）シリーズのL2記事
- 製造ライン全体を俯瞰するカタログ記事
- カテゴリ：インフォメーション（未登録者も閲覧可）

### 前提知識（記事内で使ってよいシリーズ用語）

- **延命医**: Web標準への準拠度と長期生存根拠を診断する視点
- **不自然な構造（悪）**: 採用しなかった技術の備忘録
- **製造ライン**: ClaudeMixを構成するサービス・ツールの組み合わせ全体

### 構成

```markdown
### ラインのコンセプト

このスタックで防ぐ技術的負債を1-2段落で説明します。
延命医の視点から、なぜこの技術の組み合わせが「長期生存」につながるのかを述べます。

### 標準採用リスト

### ポイント

- **選定種別**: 独立選択 / 連動（○○連動） / 連合選択 / 事実上の標準 の4種で分類
- **延命医の視点**: 「なぜ長期間生き残れるか」の根拠を一言で示す
- **不自然な構造（悪）**: 敗因・不採用理由を正直に記録（読者の判断材料になる）
- **L3リンク**: 初稿時は「後で追加」とし、L3記事公開後にリンクを差し替える

## テンプレート4: L3 技術詳細（教本）

### 対象記事

- 生存戦略スタック・インデックス シリーズのL3記事（技術ごと）
- カテゴリ：ClaudeMix ガイド（ログイン必須・有料）

### 本文構成

````markdown
### 延命医の診断

この技術をどのような視点で評価・採用したかを2-3文で示します。
Web標準への準拠度と長期生存根拠を中心に記述します。

### 選定理由

#### {選定種別}の根拠

（連動選択の場合）起点となった技術制約と、この技術が解決した問題を説明します。
（連合選択の場合）互いが前提になっている構造的必然性を説明します。
（独立選択の場合）他の選択肢と比較した際の決定要因を説明します。
（事実上の標準の場合）デファクトスタンダードになっている根拠を説明します。

#### なぜ代替案を選ばなかったか

代替案と不採用理由を明示します（不自然な構造（悪）として記録）。

### 構造的メリット

この技術を採用したことで得られた、設計上の恩恵を3点以内で説明します。

1. **{メリット1}**: {説明}
2. **{メリット2}**: {説明}
3. **{メリット3}**: {説明}

### ClaudeMixでの活用例

実際のコード・設定例を示します。

```typescript
// 実際に動作するコードスニペット
```

### 公式リファレンス・更新履歴

- [公式ドキュメント]({URL})
- 最終確認日: {YYYY-MM-DD}
- 次回確認予定: {メジャーバージョンアップ時}
````

### ポイント

- **選定種別に応じた選定理由**: 選定種別（連動/連合/独立/事実上の標準）で説明の軸が変わる
- **代替案の記録**: 不採用の技術を「不自然な構造（悪）」として正直に残す
- **構造的メリット**: 機能の説明ではなく「なぜ設計が良くなるか」を書く
- **公式リファレンス**: URLと確認日を必ず記載（情報の鮮度を保つ）

### L3記事の執筆知見（過去の判断例）

L3記事を書くとき、以下の実例を参考に自分の判断を組み立てること。ルールとして従うのではなく、「こういう考え方があった」という視点として使う。

---

**【例: 延命医の診断の書き方】**

D1記事では最初「SQLiteベースで長期生存が見込めます」と書いただけだった。これは見出しと同義で根拠がない。そこでSQLite誕生（2000年）という歴史的実績・Cloudflareの継続投資・Edge DB市場の拡大という3軸を並べた。結果、読者に「これは流行ではなくインフラだ」という感覚を与えることができた。何を「根拠」として選ぶかは技術によって変わるが、定性的な感想だけでは弱いという判断は共通して使える。

---

**【例: 代替案比較の書き方】**

D1記事のSupabase不採用理由を最初「IP制限ができない」「Hyperdriveでコストが増える」という現象面だけで書いた。これはSupabase + Hyperdriveという反論に弱い。そこで「WorkersからSupabaseへはTCP DB経由のHTTP呼び出しが必要になる。D1はRuntime Bindingで直結するため、接続モデルが根本的に異なる」という構造的な違いを先に置いた。現象はその結果として説明する順序にすると、反論されにくい記事になる。

---

**【例: コード例の書き方】**

D1記事で「ブログ記事データをD1に保存する例」を書こうとしたが、実際はリポジトリ管理だった。事実と異なるコード例を書くと記事全体の信頼性が落ちる。コードベースで実際の使用ドメインを確認してから書くという判断をした。

---

**【例: BAB Beforeの組み立て方】**

D1記事のBeforeを最初「どのDBを選べばいいか迷い始めた」という抽象描写にしていた。その後、実際にWeb上にある「Workers + SupabaseでHyperdriveが必要になった開発者」「IPレンジ固定できずに詰まった開発者」という2つのパターンを組み合わせてBeforeを再構成した。読者が「これは自分だ」と感じる解像度が上がった。架空の悩みより実在するパターンの方が刺さる。

---

**【例: 末尾フックの作り方】**

D1記事の末尾を最初「次の記事ではKVを解説します」という案内型にしていた。これは読者が止まる。「D1でデータは保存できる。しかし本番サービスにはセッション・認証メール・課金状態が必要でD1だけでは完成しない」という不完全感に変えた。読者が「次を読まなければ」と感じる構造になった。

---

## 共通ルール（すべてのテンプレート）

### 文体

- ですます調で統一
- 文のまとまりで適度に改行を入れる
- 専門用語は初出時に説明を加える

### フォーマット

- 横線（`---`）はフロントマターとの区切り以外では使用しない
- コードブロックには必ず言語指定（例: ` ```typescript `）
- 見出しは `###` (h2) から `#####` (h4) を使用
- `###` (h2) の見出しは記事全体で10個程度に収める

### コード

- **再現性**: 実際に動作する、再現性のあるコードを掲載
- **コメント**: 重要な部分にはコメントで説明を追加
- **diff形式**: 変更箇所が明確になるよう推奨

### リンク

- リンクは必要最小限に留める
- 記事の主題を補足する参考リンクのみ掲載
- 外部リンクには説明を追加（何のリンクか分かるように）

### 最後の一文

記事の最後には、読者への問いかけを含めてください：

**標準テンプレート:**

```markdown
同じような課題で悩んだ方はいませんか？
もっと良い解決方法があれば教えてください！
```

**トラブルシューティングテンプレート:**

```markdown
同じような問題で困っている方の参考になれば幸いです！
```

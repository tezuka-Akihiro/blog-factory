# Phase 1: 入力検証

オペレーターが `$ARGUMENTS` として渡した情報を評価する。
**不足があれば執筆を開始せず、補足を求めて停止すること。**

## 入力形式

以下の3形式をすべて受け付ける。どの形式かを最初に判定する。

| 形式 | 説明 | 例 |
| :--- | :--- | :--- |
| **A: 生ポスト** | Xのポスト本文をそのまま貼り付けたもの | 「知り合いの代理店が…MCCが乗っ取られて…」 |
| **B: 要約＋URL** | トレンドの概要とソースURLを組み合わせたもの | 「Claude 4発表。公式: https://...」 |
| **C: A＋補足** | 生ポストに加えてオペレーターの追記があるもの | 生ポスト＋「バイブコーダーへの影響を書いてほしい」 |

## 評価手順

形式を判定した上で、以下の3指標を確認する。

| 指標 | 判定条件 | 形式Aの場合の扱い |
| :--- | :--- | :--- |
| **トレンドの核心** | 「何が」「どう変わった/起きた」が読み取れる | ポスト本文から抽出する |
| **一次情報の所在** | 出典が確認できる | ポスト本文自体を一次情報として扱う |
| **読者への接点** | バイブコーダー・技術者・個人事業主への影響が想像できる | ポストの内容から推定して可とする |

**形式A（生ポスト）の場合:** ポスト本文がそのまま一次情報になるため、URLがなくても「一次情報の所在」は満たしているとみなす。ただし、ポスト内容からトレンドの核心と読者への接点が読み取れることが条件。

## 判定と対応

### ✅ 3指標すべて満たす → Phase 2 へ進む

入力例（十分・形式A 生ポスト）:

> 「知り合いの代理店がとあるAI導入したらMCCが乗っ取られて桁違いの損害でてて震えた。こういうのこれから増えそうですね。（中略）MCCが正規権限で操作できる状態になっている場合…」

入力例（十分・形式B 要約＋URL）:

> 「Anthropicが Claude 4 を発表。コンテキスト200万トークン、エージェント機能が大幅強化。公式: https://... バイブコーダーの開発フローへの影響を書いてほしい。」

入力例（十分・形式C A＋補足）:

> 「[生ポスト貼り付け] ← この件でバイブコーダーが気をつけるべきことを記事にしてほしい。」

### ❌ 不足がある → 補足を求めて停止

**トレンドの核心が読み取れない場合（ポストが断片的すぎる）:**
> 「ポストの内容からトレンドの核心が読み取れませんでした。何が起きた/変わったのかを1文で補足してもらえますか？」

**読者への接点が不明な場合（話題がClaudeMix読者と無関係に見える）:**
> 「このトレンドはバイブコーダー・技術者・個人事業主にどう関係しますか？記事の方向性を教えてください。」

入力例（不足）:

> 「Gemini が話題になってる。記事書いて。」
> → 返答: 「どのような発表・出来事でしょうか？ポスト本文か、概要と出典URLを共有してください。」

## 完了条件

3指標をすべて満たしていることを確認したら、以下を簡潔にまとめてオペレーターに提示し、次のフェーズへ進む。

```
【入力確認】
- 入力形式: [A: 生ポスト / B: 要約＋URL / C: A＋補足]
- トレンドの核心: [1文でまとめた内容]
- 一次情報: [ポスト本文 / URLまたは発表元]
- 記事の方向性: [読者への接点]

→ Phase 2（AI分析）に進みます。
```
